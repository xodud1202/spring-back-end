## spring-back-end 도메인 구조

### 1) 현재 코드 구조
- `src/main/java/.../controller`: 컨트롤러를 관리한다.
- `src/main/java/.../service`: 서비스 계층을 관리한다.
- `src/main/java/.../mapper`: MyBatis 매퍼를 관리한다.
- `src/main/java/.../domain`, `entity`, `repository`, `common`: 도메인 객체와 저장 계층, 공통 코드를 관리한다.

### 2) 컨트롤러 규칙
- 컨트롤러는 `프로젝트 + 도메인` 단위로 생성한다.
- 기능 단위 임시 컨트롤러나 중복 컨트롤러는 만들지 않는다.
- 동일 도메인의 API는 가능한 한 동일 도메인 컨트롤러에서 관리한다.

### 3) 도메인 책임 관리 원칙
- `Service`, `Mapper`, `Domain(VO/PO/DTO)`, `Support`, `Test`는 모두 도메인 책임 기준으로 파일을 관리한다.
- 신규 메서드나 쿼리를 작성하기 전에 해당 기능의 주 책임 도메인을 먼저 판단한다.
- 같은 도메인이면 기존 도메인 파일에 추가한다.
- 다른 도메인이면 기존 큰 파일에 누적하지 말고 해당 도메인 전용 파일을 새로 생성한다.
- 아래 항목은 고정 목록이 아니라 도메인 분리 방식의 예시다.
  - 쿠폰 금액 계산 메서드: `CouponService`
  - 주문 취소 금액 계산 메서드: `OrderCancelService`
  - 배송비 계산 메서드: `DeliveryFeeService`
  - 상품 이미지명 또는 이미지 경로 조회 메서드: `GoodsImageService`
- 이후 포인트, 정산, 리뷰, 반품, 교환, 장바구니, 마이페이지, 배송추적 등 새 도메인이 생기면 동일 기준으로 전용 파일을 만든다.

### 4) 계층별 적용 규칙
- `Service`
  - 서비스 메서드는 반드시 해당 도메인 책임 기준으로 배치한다.
  - 다른 도메인 기능이면 해당 도메인 전용 `XxxService`를 생성하거나 기존 도메인 서비스로 이동한다.
  - 공통 테이블 조회 결과를 화면/기능별 응답 객체로 변환하는 책임은 서비스에 둔다.
- `Mapper`
  - 매퍼 쿼리도 반드시 해당 도메인 책임 기준으로 배치한다.
  - 다른 도메인 쿼리를 하나의 매퍼에 혼합하지 않는다.
  - 다른 도메인 쿼리가 섞여 있으면 해당 도메인 전용 `XxxMapper`로 이동하거나 분리한다.
  - `SITE_INFO` 같은 공통 테이블은 기능별 전용 조회를 반복 작성하지 말고 공통 매퍼에서 재사용한다.
  - `CART` 테이블 쿼리는 `CartMapper`로 고정하고 다른 매퍼에 혼합하지 않는다.
- `Domain(VO/PO/DTO)`
  - 도메인 객체도 사용처 기준이 아니라 책임 도메인 기준으로 관리한다.
  - 재사용 가능성이 있는 보조 타입은 inner class로 두지 말고 별도 파일로 분리한다.
  - 공통 테이블 응답 객체는 기능별 최소 컬럼 VO를 여러 개 만들기보다 공통 확장 가능한 VO를 우선 검토한다.
- `Support`
  - 도메인 보조 클래스는 해당 도메인 하위 패키지에 둔다.
- `Test`
  - 테스트도 운영 코드와 동일한 도메인 경계를 따른다.

### 5) 리팩토링 원칙
- 작업 중 다른 도메인의 메서드, 쿼리, DTO, 보조 클래스가 섞여 있으면 현재 작업 범위에서 함께 정리하는 것을 기본으로 한다.
- 단순 위임만 분리하고 실제 구현을 기존 큰 파일에 남겨두는 방식은 지양한다.
- 도메인 분리 후에는 외부 진입점뿐 아니라 내부 계산 메서드, 보조 메서드, 보조 타입도 가능한 한 함께 이동한다.
- `SITE_INFO`와 `CART`처럼 여러 기능에서 재사용되는 테이블은 중복 조회를 새로 추가하지 않고 기존 공통 매퍼/공통 VO로 흡수하는 것을 기본 원칙으로 한다.
